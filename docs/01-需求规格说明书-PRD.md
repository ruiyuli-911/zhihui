## 一、项目概述

### 1.1 项目背景

大龄劳动者（45+/50+）就业难的核心问题是 **人、岗、政三方信息割裂**。传统线上平台操作复杂、适老性差，线下中介数字能力弱、效率低。本项目旨在通过微信小程序 + AI 语音交互 + 线上线下结合的模式，构建一个聚焦大龄劳动者的智能化数字就业服务平台。

### 1.2 项目目标

| 阶段 | 时间 | 目标 |
|------|------|------|
| **第一期（MVP）** | 2026年7月7日 | 上线核心闭环，覆盖山阳县试点 |
| 第二期 | 2026下半年 | 扩展商洛-西安区域，签到防伪升级，Web管理后台 |
| 第三期 | 2027-2028 | 陕西省内推广，工资争议处理完善，决策报告 |

### 1.3 第一期核心指标

| 指标 | 目标值 |
|------|--------|
| 覆盖区域 | 山阳县 |
| 注册求职者 | 5000人 |
| 合作企业 | 50家 |
| 带动就业 | 4700人 |
| 上线时间 | 2026年7月7日 |

### 1.4 用户规模预估

| 维度 | 数值 |
|------|------|
| C端求职者 | 5000人 |
| B端企业 | 50家（50个管理员账号） |
| G端/Admin | 3-5人（李蕊玉分配账号） |

## 二、用户角色与权限体系

### 2.1 账号体系总览

采用 **账号（Account）与档案（Profile）分离** 设计：

```
accounts（微信账号主体）
   ├── 平台管理员（李蕊玉团队）
   ├── 政府工作人员（就业部门）
   ├── 企业管理员（每企业1人，绑微信）
   └── 求职者（微信一键登录）
            │
            ├── jobseekers（求职者档案）
            │     ├── 本人档案
            │     └── 代注册档案（村干部/子女帮注册）
            │
            └── company_admins（企业管理员映射）
```

### 2.2 角色定义

| 角色 | 标识 | 权限范围 | 登录方式 | 终端 |
|------|------|---------|---------|------|
| **求职者** | C端 | 浏览岗位、报名、签到、查看工资 | 微信一键登录 / 手机号验证码 | 微信小程序 |
| **企业管理员** | B端 | 发布岗位、管理报名、扫码签到、录入工资 | 微信一键登录（绑定企业） | 微信小程序 |
| **政府工作人员** | G端 | 查看辖区数据看板、政策发布、企业监管 | 账号密码（李蕊玉分配） | 微信小程序（管理端） |
| **平台管理员** | Admin | 审核企业、管理用户、系统配置 | 账号密码（李蕊玉分配） | 微信小程序（管理端） |

### 2.3 数据权限规则

| 角色 | 可见数据范围 |
|------|------------|
| 求职者 | 仅本人数据 |
| 企业管理员 | 本企业岗位/报名/签到/工资数据 |
| 政府工作人员 | **仅管辖区域（areaIds）** 的聚合数据，不可查看个人手机号/身份证 |
| 平台管理员 | 全平台数据 |

> **关键约束**：政府端默认只能看脱敏聚合数据。如需查看求职者联系方式，需单独授权并记录日志。

## 三、功能需求详述

### 3.1 C端 — 求职者端（微信小程序）

#### 3.1.1 登录注册

| 功能 | 详细描述 |
|------|---------|
| **微信一键登录** | 首次授权微信手机号，自动创建账号 + 求职者档案 |
| **手机号验证码登录** | 作为备选方案 |
| **他人代注册** | 村干部/子女帮大龄劳动者注册，填写：姓名、性别、年龄、技能、电话（可选）、期望工种。代注册关系记录 `registeredBy` 字段 |
| **信息补全** | 首次登录引导填写：年龄、性别、技能标签（建筑/保洁/搬运等）、期望工种、期望区域 |

#### 3.1.2 语音交互（适老化核心特色）

| 功能 | 详细描述 |
|------|---------|
| **语音入口** | 首页底部悬浮橙色麦克风按钮（64px），始终可见 |
| **语音→文字识别** | 集成微信同声传译插件，实时语音转文字 |
| **语音导航** | 用户说"找工作"→跳转岗位列表；"建筑工"→自动筛选；"我的报名"→跳转报名记录 |
| **语音播报** | 关键信息（岗位名称、薪资）自动朗读 |
| **操作确认** | 语音仅做导航，**所有写操作（报名/取消/确认）必须手动按钮点击** |
| **失败兜底** | 识别失败→展示文字引导按钮；连续失败→展示客服电话 |
| **前匹配** | 关键词（"找工作""工资""报名"）在前端本地匹配，不调云函数 |

#### 3.1.3 岗位浏览

| 功能 | 详细描述 |
|------|---------|
| **岗位列表** | 卡片式展示：岗位名称（18px Bold）、企业名称、薪资（22px Bold 橙色）、距离、工种标签 |
| **岗位筛选** | 按工种、薪资范围、区域筛选 |
| **岗位搜索** | 支持语音搜索 + 文字搜索 |
| **岗位详情** | 薪资待遇、工作时长、食宿情况、岗位要求、企业信息、地址、联系人 |
| **收藏岗位** | 心形图标收藏，便于后续查看 |

#### 3.1.4 报名流程

| 功能 | 详细描述 |
|------|---------|
| **一键报名** | 岗位详情页点击"立即报名" |
| **报名前确认** | 确认个人基本信息（姓名、电话），可修改 |
| **报名成功提示** | 成功页展示，告知"等待企业联系" |
| **报名记录** | "我的报名"列表，状态：已报名 / 已录取 / 未通过 / 已取消 |
| **取消报名** | 仅"已报名"状态可取消 |
| **完成确认** | 工作结束后，劳动者点击"确认完成" |

#### 3.1.5 签到系统（MVP上线）

| 功能 | 详细描述 |
|------|---------|
| **一人一码** | 每个求职者生成专属微信小程序码（使用 `wxacode.getUnlimited`，免费） |
| **签到方式** | **劳动者展示二维码 → 企业扫码签到**，劳动者无需操作手机 |
| **防伪机制** | 码内含用户ID+时间戳，云函数校验：是否已被该企业录取、是否当日已签到、岗位是否有效 |
| **签到记录** | 查看历史签到明细（日期、岗位、企业、签到时间） |

> 二维码方案安全说明：微信小程序码只能用微信扫描，码内参数加密无法篡改，扫码时获取企业管理员 openid 做身份校验。MVP暂不做动态刷新和地理位置校验，第二期升级。

#### 3.1.6 工资管理（MVP上线）

| 功能 | 详细描述 |
|------|---------|
| **工资单查看** | 查看每笔工资记录（工资期间、企业申报金额、确认状态） |
| **工资确认** | 劳动者确认金额无误后点击"确认收到"（仅记录确认，不涉及资金流转） |
| **工资申诉** | 对金额有异议可发起申诉，填写原因，平台介入处理 |
| **状态标识** | 严格区分：🟡 `企业申报` → 🟢 `已确认` → 🔴 `有异议` |

> **重要**：界面文案严格区分"企业申报"和"用户确认"，避免"企业录入=已支付"的法律风险。

#### 3.1.7 通知机制

| 通知类型 | 方式 | 说明 |
|---------|------|------|
| 企业录取通知 | **短信** | 企业录取后，自动发短信通知劳动者 |
| 工资录入通知 | **短信** | 企业录入工资后，短信提醒劳动者确认 |
| 日常通知 | 微信订阅消息（后续加） | 替代部分短信，降低成本 |
| 人工电话 | 电话 | 李蕊玉团队对重点用户电话通知 |

#### 3.1.8 个人中心

| 功能 | 描述 |
|------|------|
| 个人信息 | 姓名、头像、电话、年龄、技能标签、期望工种 |
| 我的报名 | 报名记录列表（含状态） |
| 我的签到 | 签到记录列表 |
| 我的工资 | 工资单列表 |
| 我的收藏 | 收藏岗位列表 |
| 设置 | 语音播报开关 |

## 四、用户核心流程（含状态机）

### 4.1 求职者核心路径

```
打开小程序 → 微信一键登录 → 补全求职者档案
    ↓
首页（语音可操作 / 手动浏览）
    ↓
【语音】"我要找工作" → 跳转岗位列表
    ↓
筛选/搜索 → 查看岗位详情
    ↓
点击"立即报名" → 确认信息 → 报名成功
    ↓                           状态: submitted
等待企业审核
    ├── 企业录取 → 短信通知      状态: accepted
    └── 企业拒绝                 状态: rejected
           ↓
到岗 → 劳动者展示二维码
    ↓ 企业扫码签到                状态: checked_in
工作完成 → 劳动者确认完成         状态: completed
    ↓
企业录入工资 → 短信通知劳动者
    ↓
劳动者确认工资 / 发起异议
```

### 4.2 报名状态机

```
submitted（已报名）
    ├── cancelled（已取消） ← 用户主动取消
    └── accepted（已录取） ← 企业录取
            ├── completed（已完成） ← 用户确认工作完成
            └── rejected（未通过） ← 企业拒绝
```

**关键约束**：
- 同一求职者不可重复报名同一岗位
- 报名时需校验：用户是否登录→岗位是否审核通过→岗位是否招聘中→是否重复报名
- 报名和岗位人数统计更新放在同一事务中

### 4.3 企业核心路径

```
打开企业端 → 微信一键登录 → 绑定企业（首次）→ 完善信息→ 上传营业执照
    ↓                                         状态: pending
等待平台审核
    ├── 审核通过                               状态: approved
    └── 审核拒绝 → 修改后重新提交              状态: rejected
           ↓
发布岗位 → 等待审核（可选开关）
    ↓
查看报名列表 → 录取求职者 → 短信通知
    ↓
劳动者到岗 → 扫码签到
    ↓
录入工资 → 等待劳动者确认
```

### 4.4 政府端核心路径

```
打开管理端 → 账号密码登录
    ↓
首页数据看板（辖区就业数据总览）
    ↓
查看分析 / 监管企业 / 发布政策
    ↓
数据导出（Excel）
```

### 4.5 Admin核心路径

```
打开管理端 → 账号密码登录
    ↓
待审核企业列表 → 审核通过/拒绝
    ↓
待审核岗位列表 → 审核通过/拒绝
    ↓
数据总览 / 用户管理 / 系统设置
```

## 五、核心数据实体

### 5.1 账号与用户体系

```
accounts（微信账号）
├── 求职者 → jobseekers（求职者档案）
│              └── 可被代注册（registeredBy 记录代注册人）
├── 企业管理员 → company_admins（企业管理员映射）
├── 政府人员 → admins（管理账号）
└── 平台管理员 → admins（管理账号）
```

**accounts**
```js
{
  _id, openid, phoneMasked, role, status, createTime
}
```

**jobseekers**
```js
{
  _id, accountId, name, phoneCipher, phoneHash, phoneMasked,
  gender, birthYear, skills[], expectJob, expectArea,
  registeredBy,      // 代注册人accountId（可选）
  relation,          // "子女"|"村干部"|"工作人员"
  status, createTime, updateTime
}
```

**company_admins**
```js
{
  _id, companyId, accountId, role: 'admin',
  status, createTime
}
```

**admins**（平台/政府管理员）
```js
{
  _id, account, passwordHash, name,
  role: 'platform_admin' | 'gov_admin',
  areaIds[],            // 管辖区域（政府人员用）
  status, createTime, updateTime
```

## 八、非功能需求
