# 智慧小职 — 架构修订方案 v2

> 日期：2026-07-03  
> 基于反馈重构：三层意图路由 / 可中断状态机 / 安全校验 / 适老交互

---

## 一、当前架构的问题

| 问题 | 影响 |
|------|------|
| 单层关键词意图匹配太脆弱 | 用户换种说法就掉到 small_talk |
| 状态机线性不可中断 | 报名流程中问"包住吗"会卡死 |
| apply_job 缺幂等性 | 重复点击可能重复报名 |
| 用户资料无校验 | "今年200岁"也能入库 |
| RAG 缺来源展示 | 用户不知道答案从哪来的 |
| 语音无确认机制 | ASR 识别错了直接执行 |

---

## 二、三层意图路由（替代单层匹配）

```
第一层：强规则拦截
  ─ 报名岗位 J1003          → apply_job_by_id
  ─ 报名第一个 / 第二个      → apply_job_by_index
  ─ 确认 / 取消             → confirm / cancel
  ─ 1 开头的 11 位数字      → provide_info (phone)
  ─ 我叫XX / 今年X岁        → provide_info

第二层：轻量语义分类（关键词+TF-IDF）
  ─ 找工作 / 岗位 / 附近    → search_job
  ─ 社保/工伤/合同/欠薪     → policy_query
  ─ 进度/记录/状态          → application_status
  ─ 你好 / 在吗             → greeting
  ─ 详情 / 怎么样           → job_detail

第三层：LLM 兜底分类（仅输出 JSON，不执行业务）
  输入: 用户原始文本
  输出: {"intent": "search_job", "slots": {...}, "confidence": 0.91}
  限制: 置信度低于 0.6 走 small_talk
```

## 三、状态机重构：主任务 + 上下文分离

### 主任务状态（线性）

```
IDLE → SEARCH_RESULT → WAITING_CONFIRM → COLLECTING_PROFILE → SUBMITTING → SUCCESS
```

### 上下文状态（持久化，不随主任务重置）

```
last_jobs: [...]            # 上次搜索结果
selected_job: {...}          # 当前选中的岗位
pending_action: "confirm"    # 待执行操作
missing_fields: [...]        # 缺哪些资料
last_intent: "search_job"    # 上一条意图
rag_history: [...]           # RAG 问答记录
```

### 允许中断的操作

任何时候用户都可以：

| 操作 | 行为 |
|------|------|
| 查岗位详情 | 保持主状态，回答后提示继续 |
| 政策咨询 | 保持主状态，回答后提示继续 |
| 闲聊 | 保持主状态 |
| 重新搜索 | 重置主状态到 SEARCH_RESULT |
| 取消 | 重置主状态到 IDLE |

## 五、用户资料校验

| 字段 | 校验规则 |
|------|---------|
| 姓名 | 2-10 位中文或字母，无特殊字符 |
| 年龄 | 16-80 之间的整数 |
| 电话 | 1 开头 11 位数字 |

```python
def validate_info(data: dict) -> list[str]:
    errors = []
    if "name" in data:
        if len(data["name"]) < 1 or len(data["name"]) > 10:
            errors.append("姓名长度不对")
    if "age" in data:
        if not (16 <= data["age"] <= 80):
            errors.append("年龄需要在 16-80 之间")
    if "phone" in data:
        if not re.match(r'^1\d{10}$', str(data["phone"])):
            errors.append("手机号格式不对")
    return errors
```

## 六、RAG 三阶段规划

| 阶段 | 检索方式 | 特点 |
|------|---------|------|
| 一（当前） | TF-IDF + 关键词 | 15 篇文档，演示够用 |
| 二 | BM25 + 向量 + Rerank | 召回率更高，需要 embedding |
| 三 | 混合检索 + 置信度阈值 | 低于阈值不回答，引导追问 |

### 返回结构

```json
{
  "type": "knowledge_answer",
  "reply": "根据劳动合同相关规定……",
  "sources": [{"title": "劳动合同注意事项", "score": 0.86}],
  "confidence": "high"
}
```

## 七、语音交互双模式

### 模式一：按住说话（复杂任务）

```
按住 → 录音 → 松开 → 展示识别文字 → 用户确认 → 执行
```

### 模式二：一键指令（固定操作）

```
[找工作] [报名] [查进度] [语音咨询]
```

### 确认机制

```
用户说："找高新区四千以上包住的保安"
页面显示：
  岗位：保安 | 地点：高新区 | 工资≥4000 | 包住
  [对，就是这样] [我想改一下]
用户确认后再检索
```

## 九、业务 vs LLM 边界

| 能力 | 是否可由 LLM 决定 |
|------|-----------------|
| 报名是否执行 | ❌ 不可以 |
| 选哪个岗位 | ❌ 不可以 |
| 查谁的数据 | ❌ 不可以 |
| 岗位信息 | ❌ 不可以（必须工具返回） |
| 政策依据 | ❌ 不可以脱离检索 |
| 识别口语表达 | ✅ 可以 |
| 提取筛选条件 | ✅ 可以 |
| 回复润色 | ✅ 可以 |
| 情绪安抚 | ✅ 可以 |

## 十、MVP 实施顺序

```
第一步：真实岗位查询 + 岗位卡片 + 搜索条件提取  ← 当前阶段
第二步：选岗位 + 确认报名 + 补资料 + 防重复报名
第三步：语音转文字 + 识别结果确认
第四步：RAG 政策问答 + 来源展示
第五步：TTS 朗读 + 适老化交互优化
第六步：面试提醒、到岗确认、导航
```
